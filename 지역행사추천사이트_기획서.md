# 지역 행사 추천 사이트 기획서 (프로토타입)

## 1. 프로젝트 개요

| 항목 | 내용 |
|---|---|
| 서비스명(가안) | 우리동네 행사 (가칭) |
| 목적 | 나이·동반자·관심사 정보를 바탕으로, 내가 실제로 참여 가능한 지역 행사를 추천해주는 서비스 |
| 문제의식 | 지역에 좋은 행사가 많지만 흩어져 있어 찾기 어렵고, "이 행사가 나(혹은 우리 가족)에게 맞는지"를 판단하기 어려움 |
| 초기 데이터 제공 범위 | 서울 강북권(노원구, 도봉구 — 방학동 등) 한정 |
| 지역 선택 UI 범위 | 전국 단위로 넓게 제공 (확장을 고려한 구조), 단 현재는 강북권 외 지역 선택 시 "준비 중" 안내 |
| 로그인 | 없음 (프로토타입 단계에서는 정보 입력만으로 시작, 로그인/회원가입 미포함) |
| 기술 스택 | React (Vite), 상태관리는 useState/useReducer, 스타일은 CSS Module 또는 Tailwind |

## 2. 온보딩(정보 입력) 화면 — 최종 항목

로그인 없이 진입 시 바로 받는 정보입니다. 대화 중 논의한 내용을 반영해 **필수 항목**과 **선택 항목(건너뛰기 가능)**으로 나눴습니다.

### 필수 항목
1. **본인 나이대**: 영유아 / 어린이 / 청소년 / 20-30대 / 40-50대 / 60대 이상
2. **사는 지역**: 시/도 → 구/군 → 동 순으로 드롭다운 선택 (전국 지원)
   - 강북권(노원구/도봉구) 외 지역 선택 시: "아직 이 지역 정보는 준비 중이에요. 노원·도봉 지역 정보부터 만나보세요" 안내 후 강북권 전환 유도
3. **동반자 여부**: 있음 / 없음
   - "있음" 선택 시 → **동반자 나이대 멀티 선택** (본인과 동일한 6개 구간 중 중복 선택 가능, 예: 60대 이상 + 어린이 = 조부모-손자 동반)
4. **관심 카테고리** (멀티 선택, 최소 1개): 문화공연 · 전시/체험 · 벼룩시장/마켓 · 스포츠/건강 · 교육/강좌 · 축제

> 관심 카테고리는 나이·지역만으로는 추천이 너무 뭉뚱그려지기 때문에 필수로 배치했습니다.

### 선택 항목 (건너뛰기 가능 — "나중에 설정" 버튼 제공)
5. 선호 요일/시간대: 평일 낮 / 평일 저녁 / 주말
6. 비용 선호: 무료만 / 유료 포함
7. 이동 수단: 도보 / 대중교통 / 자차
8. 반려동물 동반 여부
9. 거동 불편·유모차 접근성 필요 여부

> 선택 항목은 초기 진입 장벽을 낮추기 위해 스킵 가능하게 하고, 추후 "내 정보 수정"에서 채울 수 있게 합니다.

## 3. 거리 필터 — 제안 조정

말씀하신 "근처 / 100km / 200km" 단위는 전국 서비스라면 맞지만, **강북 특정 구 단위로 시작하는 프로토타입에는 너무 넓은 단위**라 실제 체감 차이가 없을 수 있어요. 아래처럼 조정하는 걸 제안드립니다.

| 단위 | 의미 |
|---|---|
| 도보 거리 (~1km) | 걸어서 갈 수 있는 동네 행사 |
| 우리 동네 (같은 동/인접 동) | 생활권 내 행사 |
| 우리 구 (노원구 또는 도봉구 전체) | 구 단위 행사 |
| 인근 구까지 | 강북권 인접 구 포함 (성북구, 중랑구 등 추후 확장 시) |

지역 확장(전국 단위) 이후에는 원래 제안하신 "근처/100km/200km" 같은 광역 단위도 다시 의미가 생기니, 그건 **Phase 4(전국 확장)** 때 다시 설계하는 걸 추천드려요.

## 4. 개발 단계(로드맵)

### Phase 1 — MVP (정보 입력 + 추천 리스트)
- 온보딩 폼 (위 2번 항목 기준)
- 강북권(노원·도봉) 행사 데이터 하드코딩/수동 입력으로 20~30개 정도 확보해 프로토타입 시연
- 나이대 + 관심 카테고리 기준으로 필터링된 추천 리스트 표시
- 거리 필터: 도보/우리 동네/우리 구 3단계

### Phase 2 — 데이터 자동화
- 서울 열린데이터광장 문화행사정보 API 연동 (자치구=노원구/도봉구 필터링)
- API로 못 채우는 부분은 관리자 수동 등록으로 보완
- 데이터 갱신 주기 설정 (일 1회 등)

### Phase 3 — 추천 고도화
- 동반자 나이대 조합을 고려한 추천 로직 고도화 (예: 60대+어린이 동반 조합에 맞는 행사 가중치)
- 선택 항목(요일/비용/이동수단/반려동물/접근성) 반영한 세부 필터
- 행사 상세 페이지 (지도, 대중교통 안내, 문의처)

### Phase 4 — 지역 확장
- 강북권 외 지역으로 데이터 확장 (지자체별 API 조사 필요)
- 광역 거리 단위(근처/100km/200km) 재도입
- 다른 지자체 열린데이터광장 연동

> **개발 순서 추천**: Phase 1은 데이터 없이도 UI/추천 로직을 검증할 수 있도록 목업 데이터로 먼저 완성하고, Phase 2에서 실제 API 연동을 붙이는 순서를 권장합니다.

## 5. 화면 구성

### 5.1 온보딩 화면

한 화면에 모든 항목을 욱여넣기보다, 위에서 아래로 자연스럽게 흐르는 세로 스크롤 방식의 단일 페이지로 구성하는 걸 추천드려요. 진입하자마자 폼부터 보여주기보다, 맨 위에 서비스가 무엇을 해주는지 한 줄로 짧게 전달하는 문구를 두고(예: "나와 우리 가족에게 맞는 동네 행사를 찾아드려요") 바로 아래에 필수 입력 4가지(나이대, 지역, 동반자, 관심 카테고리)를 순서대로 배치합니다.

나이대는 버튼형 선택지(칩 형태)로 6개를 나란히 두어 한 번의 탭으로 고르게 하고, 지역은 시/도 → 구/군 → 동 순서로 단계적으로 펼쳐지는 방식이 좋아요 — 처음부터 3단 드롭다운을 다 보여주면 복잡해 보이니, 시/도를 고르면 그 아래 구/군 선택지가 나타나고, 구/군을 고르면 동 선택지가 나타나는 식으로 단계적으로 노출하는 게 훨씬 깔끔합니다.

동반자 여부는 "있음/없음" 두 개의 큰 버튼으로 먼저 묻고, "있음"을 선택했을 때만 그 아래로 동반자 나이대 칩 선택지가 부드럽게 펼쳐지도록 합니다. 관심 카테고리는 나이대와 마찬가지로 칩 형태의 다중 선택으로, 최소 1개 선택 전까지는 다음 버튼이 비활성화 상태임을 시각적으로 명확히 표시해주세요.

선택 입력(요일/비용/이동수단/반려동물/접근성) 다섯 항목은 기본적으로 접혀 있는 아코디언 형태로 두고, "더 정확한 추천을 원하시면" 같은 부드러운 문구로 펼쳐보도록 유도합니다. 이 부분이 펼쳐져 있으면 화면이 복잡해 보이니 기본값은 항상 닫힌 상태로 두는 게 좋아요.

맨 아래 "행사 추천받기" 버튼은 화면 전체 너비를 채우는 큼직한 형태로, 필수 항목이 다 채워지기 전까지는 비활성화 처리합니다.

### 5.2 추천 리스트 화면

상단에는 지금 어떤 조건으로 보고 있는지를 담백하게 한 줄로 보여주는 게 좋아요 (예: "노원구·도봉구 근처 행사 12건"). 그 아래 거리 필터와 카테고리 필터를 가로로 나란히 배치하되, 드롭다운보다는 탭이나 칩 형태로 두어 한 번의 탭으로 즉시 리스트가 갱신되는 느낌을 주는 게 좋습니다.

리스트는 카드 형태로, 한 카드당 정보 우선순위를 명확히 두세요: 행사명이 가장 크게, 그 아래 날짜·장소·비용을 한 줄로 담백하게, 그리고 카드 안에 여백을 충분히 둬서 정보가 빽빽해 보이지 않도록 합니다. 카테고리별로 아이콘이나 색상 태그를 작게 붙여서 훑어볼 때 카테고리 구분이 한눈에 되게 하면 좋아요.

가장 중요한 건 "이 행사가 나(혹은 우리 동반자 조합)에게 맞는 이유"를 카드 안에 짧은 배지 문구로 보여주는 부분이에요. 예를 들어 동반자로 "60대 이상 + 어린이"를 설정한 사용자에게는 카드 하단에 "손자·손녀와 함께 가기 좋아요" 같은 한 줄을 작은 배지 스타일로 붙여서, 왜 이 행사가 추천됐는지 사용자가 바로 납득할 수 있게 합니다. 이 배지가 이 화면에서 사실상 가장 차별화된 요소이니, 색상이나 타이포로 살짝 강조해주는 게 좋겠어요.

리스트가 비어있는 경우(조건에 맞는 행사가 없을 때)는 단순히 "결과 없음"이라고 하기보다, 어떤 조건을 풀면 결과가 나올 수 있는지 안내하는 문구를 넣어주세요 (예: "카테고리를 하나 더 선택해보시겠어요?").

### 5.3 (Phase 3) 행사 상세 화면

카드를 눌렀을 때 넘어가는 화면으로, 상단에 행사명과 대표 이미지(있는 경우), 그 아래 일시·장소·대상 연령·비용·문의처를 표 형태가 아니라 짧은 목록 형태로 정리합니다. 장소 아래에는 작은 지도를 붙이고, 지도 밑에 대중교통으로 오는 법을 한두 줄로 안내하면 좋아요.

리스트 화면과 마찬가지로 여기서도 "왜 이 행사가 나에게 맞는지"를 문구 하나로 다시 짚어주는 게 좋습니다 — 리스트에서 봤던 배지 문구를 상세 화면 상단에서 조금 더 풀어서 설명하는 식으로요 (예: "우리 가족 조합(성인 2명 + 어린이 1명)에게 잘 맞는 행사예요").

## 6. 데이터 구조 (제안)

```ts
// 사용자 프로필 (로컬 상태, 로그인 없음)
interface UserProfile {
  ageGroup: "영유아" | "어린이" | "청소년" | "20-30대" | "40-50대" | "60대이상";
  region: { city: string; district: string; dong: string };
  companions: {
    hasCompanion: boolean;
    companionAgeGroups: string[]; // 위 ageGroup 값들의 배열, 중복 선택 가능

  interests: string[]; // ["문화공연", "전시체험", ...]

  transport?: "도보" | "대중교통" | "자차";
  withPet?: boolean;
  accessibilityNeeded?: boolean;

// 행사 데이터
interface LocalEvent {
  id: string;
  title: string;
  category: string;        // 관심 카테고리 중 하나
  district: string;        // 자치구 (노원구/도봉구 등)
  dong?: string;
  location: { lat: number; lng: number; address: string };
  startDate: string;
  endDate: string;
  targetAge?: string;       // 원본 데이터의 "이용대상" 필드
  isFree: boolean;
  cost?: string;
  source: "공공API" | "수동등록";
  contact?: string;
  petFriendly?: boolean;
  accessibilityInfo?: string;
}
```

## 8. 추천(매칭) 로직 초안

Phase 1 프로토타입에서는 복잡한 알고리즘보다 **명확한 규칙 기반 필터 + 가중치 점수** 정도로 충분합니다.

1. **1차 필터(하드 필터)**: 지역(우리 동네/우리 구 범위 내) + 관심 카테고리 일치
2. **2차 점수(소프트 스코어링)**:
   - 행사 대상 연령에 본인 또는 동반자 나이대가 포함되면 +점수
   - 선택 입력을 채운 경우 요일/비용/이동수단/반려동물/접근성 일치 시 +점수
3. **정렬 기준**: 점수 높은 순 → 동점 시 날짜 임박 순

> 예시: "뽀로로 전시"는 대상 연령이 유아/어린이지만, 동반자에 "60대 이상"이 포함된 사용자에게도 "손자와 함께 가기 좋아요" 배지와 함께 노출 — 말씀하신 시나리오를 반영한 부분입니다.
